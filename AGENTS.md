# AGENTS.md

本文件是本仓库的唯一项目开发规则。后续在本项目中的分析、设计、编码、重构、自检与输出，默认都必须严格遵守本文件；如用户后续提出新要求，与本文件冲突时，以用户最新明确要求为准。

## 1. 项目定位

- 项目名称：家庭时令菜谱 App。
- 项目形态：移动端 App。
- 当前版本：`1.0 MVP`。
- 当前目标：优先完成核心页面与主流程闭环，先做可运行、可演示、可迭代的版本。

## 2. 核心功能范围

当前只围绕以下功能开发：

- 时令食材
- 菜谱
- 菜篮子
- 食材详情
- 菜谱详情

当前优先开发页面顺序：

1. 首页
2. 食材分类页
3. 食材详情页
4. 菜谱详情页
5. 菜篮子
6. 我的

以下功能在当前阶段默认不设计、不实现、不预留复杂架构：

## 4. 技术栈

### 前端

- `uni-app`
- `Vue 3`
- `NutUI`

### 后端

- `Node.js`
- `Express`

### 数据库

- `MySQL`

## 5. 总体开发原则

- 优先保证代码可运行。
- 不要过度设计。
- 不要增加无关功能。
- 不要自动引入复杂状态管理。
- 默认采用简单、稳定、易维护的实现。
- 在需求未明确前，不主动抽象多套模式、不提前建设复杂基础设施。
- 每次开发前先理解现有目录与代码，再落地修改。
- 不得伪造运行结果、接口结果、测试结论或完成状态。

## 6. 前端实现规则

- 必须使用 `Vue 3 Composition API`。
- 异步逻辑统一使用 `async/await`。
- 所有页面优先组件化实现。
- 尽量拆分组件，但不做过度拆分。
- 页面结构保持统一。
- 文件命名统一使用小写英文。
- 不生成无用代码。
- 不允许为了“以后可能用到”而提前加大段未使用逻辑。

## 7. TypeScript 与代码约束

- 不允许使用 `any`。
- 类型定义应尽量明确，接口返回、组件参数、核心业务结构都应有清晰类型。
- 变量、函数、组件命名必须语义化英文，不使用拼音缩写。
- 默认使用 2 空格缩进。
- 优先小函数、单一职责。
- 避免深层嵌套，优先提前返回。
- 不保留无意义注释、调试日志、注释掉的旧代码。
- 注释只解释必要的设计原因或业务约束，不写低价值描述性注释。

## 8. UI 与交互规则

- 风格参考蔚来。
- 整体风格：白色、简洁、轻量。
- 布局方式：卡片式布局优先。
- 内容展示：食材图片优先。
- 设计原则：移动端 App 优先。
- 页面观感：保持简洁，不堆砌信息，不做复杂视觉装饰。

UI 落地时应遵循：

- 页面信息层级清晰，首屏突出主要内容。
- 卡片间距、圆角、阴影保持克制，避免“过设计”。
- 列表、分类、详情页都应保持统一留白与视觉节奏。
- 图片优先展示时，也要兼顾弱网、加载失败、空图占位。
- NutUI 的使用应尽量贴合整体风格，避免出现明显割裂的默认样式堆叠。

## 9. 页面与组件规范

- 页面负责业务编排与数据组织，组件负责展示与局部交互。
- 相同结构或样式块出现两次及以上时，优先评估是否提取组件。
- 组件应单一职责，避免一个组件承担整个页面逻辑。
- 页面目录结构、命名方式、代码组织顺序尽量统一。
- 组件 `props`、事件、插槽应保持清晰，不做隐式耦合。

## 10. 推荐目录结构

如未初始化工程，默认按 `uni-app` 项目习惯组织；如脚手架结构不同，以实际生成结构为基础遵守等价约束。

推荐结构：

```text
src/
  pages/
  components/
  layouts/
  services/
  utils/
  types/
  constants/
  assets/
  static/
```

约定：

- `pages/`：页面级目录，按业务拆分。
- `components/`：可复用组件。
- `layouts/`：通用页面布局容器，如页面头部、内容容器等。
- `services/`：接口请求、数据适配、业务服务封装。
- `utils/`：纯工具函数。
- `types/`：公共类型定义。
- `constants/`：常量、枚举、配置映射。
- `assets/` / `static/`：静态资源。

## 11. 数据与接口规范

- 所有请求统一封装，禁止在页面中散落直接请求。
- 接口层负责：
  - 发起请求
  - 基础错误处理
  - 响应结构适配
- 页面层只消费经过适配后的业务数据。
- 对加载态、空态、错误态必须有基础处理。
- 不把后端原始字段直接污染整个前端展示层，必要时做字段映射。
- MVP 阶段接口设计优先直观、够用，不为了未来复杂场景提前设计重型抽象。

## 12. 状态管理规则

- 当前阶段不要自动引入复杂状态管理方案。
- 优先使用页面本地状态与组合式逻辑封装。
- 只有当跨页面共享状态明显增多且局部状态已难维护时，再评估引入更集中方案。
- 避免冗余状态与可推导状态重复存储。

## 13. 后端规则

- 后端使用 `Node.js + Express`。
- 路由职责清晰，按业务模块划分。
- 控制层、服务层、数据访问层尽量分离，但不做过度架构。
- 返回结构统一，错误响应风格统一。
- 不在控制器中堆积大量业务逻辑。
- 输入参数要做基础校验。
- 数据库查询以满足 MVP 为目标，先保证正确和清晰。

## 14. 数据库规则

- 数据库使用 `MySQL`。
- 表结构命名、字段命名统一使用小写英文风格。
- 先围绕 MVP 核心流程建设必要表结构。
- 避免为暂不开发功能提前建大量表。
- 涉及菜谱、食材、分类、菜篮子等实体时，字段语义必须清晰。

## 15. 性能与稳定性

- 优先控制首屏体验。
- 避免一次性加载过多无关数据。
- 图片资源按移动端场景控制大小。
- 列表数据在必要时再做分页或懒加载，不做空转优化。
- 所有主要页面都要对空数据、请求失败、重复点击做基础兜底。

## 16. 变更边界

- 仅修改当前任务所必需的文件。
- 不混入无关重构。
- 不随意删除、移动、重命名文件，除非任务明确需要。
- 不回退用户已有改动。
- 如果发现已有改动与当前任务冲突，先停下并明确冲突点。

## 17. 验证规则

- 每次改动完成后，至少做与改动范围匹配的自检。
- 自检优先包括：
  - 页面是否能正常打开
  - 关键交互是否闭环
  - 组件是否正常渲染
  - 样式是否明显错乱
  - 空态、错误态是否可用
- 若当前环境无法运行前端或后端验证，必须明确说明未验证项与原因。

## 18. 输出规则

每次完成任务时，默认说明：

- 完成了什么
- 修改了哪些关键文件
- 是否已验证，如何验证
- 是否存在剩余风险或下一步建议

## 19. 决策优先级

多种实现方案冲突时，按以下顺序决策：

1. 可运行
2. 正确性
3. 可维护性
4. 与当前项目规则一致性
5. 开发效率

- 后续实现必须默认遵守本文件，不得自行偏离技术栈与功能边界。
- 如项目初始化后出现更细的工程规范文件，可在不违背本文件目标的前提下补充细则。
