# 项目清理分阶段计划

## 原则

- 先锁边界，再修主链路，再清理残留
- 先做证据化整理，再做删除动作
- 本计划不等于立即执行清理

## 第一阶段：锁死 legacy 目录

目标：

- 明确 `admin-backend/`、`backend/` 不参与当前 MVP 开发

动作：

1. 在团队协作流程中固定“主线目录仅为 `frontend/`、`admin-frontend/`、`server/`”
2. PR 审查时默认拒绝对 `admin-backend/`、`backend/` 的业务修改
3. 如需引用 legacy 逻辑，只允许读，不允许继续加功能
4. 在后续文档中统一把 legacy 目录标成“冻结区”

产出：

- 目录边界不再反复讨论

## 第二阶段：修启动和接口联调

目标：

- 保证三条主线目录能稳定启动
- 保证后台配置到 C 端展示的闭环可验证

优先项：

1. 核对 `frontend`、`admin-frontend`、`server` 的实际启动命令与 README 是否一致
2. 清理“已废弃但仍可误用”的接口入口
3. 统一首页顶部导航/轮播图的唯一主链路：
   - 管理端只认 `/home/top-navs/:navId/hero-banners`
   - 服务端只保留对应挂载
   - 旧 `HomeHeroBannersPage` 只保留迁移提示，不再承载数据操作
4. 逐项验证 C 端服务层调用与服务端实际挂载是否一致

重点风险：

- deprecated API / 未挂载旧路由并存，后续清理前必须确认没有人工入口依赖旧链路

## 第三阶段：清未使用依赖和死代码

目标：

- 在不影响启动和联调的前提下收缩仓库噪音

优先清单：

1. `frontend` 中 React/shadcn/Tailwind 残留依赖
2. `admin-frontend` 中疑似未用依赖 `react-icons`
3. 已不再挂载的旧页面/旧路由/旧状态文件
4. `server/prisma/seed.js` 这类重复源文件

执行原则：

- 每次只清一小组
- 每组清理后立即验证启动和构建
- 没有验证，不做删除

## 第四阶段：整理 server route/service/schema

目标：

- 让统一后端结构与当前主线功能真正对齐

动作：

1. 按资源梳理 `server/src/routes/admin/*`
2. 把迁移残留接口标记清楚
3. 评估是否需要补 service 层边界，减少 route 文件内业务堆叠
4. 对照 `docs/backend/api-spec.md` 更新接口实际情况
5. 对照 `docs/backend/database-schema.md` 更新模型职责说明

注意：

- 本阶段是结构整理，不改数据库结构
- `server/prisma/schema.prisma` 只允许在迁移方案确认后再动

## 第五阶段：确认数据库迁移方案

目标：

- 在清楚当前模型优先级后，再决定数据库如何收口
- 2026-07-03 已先执行低风险查询索引优化；字段/表收口仍需单独评估

动作：

1. 明确哪些模型属于 MVP 必须
2. 明确哪些模型只是第二优先级或暂缓
3. 识别是否需要拆分 seed：
   - 基础 seed
   - 演示 seed
   - 资源接入 seed
4. 确认 `seed.ts` 作为唯一源，处理重复 seed 文件
5. 再决定是否需要未来迁移、归档、软废弃字段/表
6. 对 MVP 高频接口保留索引评审清单，避免后续 route 改动后索引失配

禁止事项：

- 在本阶段前，不要直接改 `schema.prisma`
- 不要为“清理好看”做破坏性迁移
- 已确认的非破坏性索引迁移可以先执行，但不得夹带字段删除、表删除或数据清理

已执行：

- 新增 `20260703090000_add_query_performance_indexes`，只包含 `CREATE INDEX IF NOT EXISTS`
- 覆盖 C 端公开内容列表、菜篮子、家庭成员校验、收藏/浏览历史、搜索历史、菜价记录等 MVP 高频读路径

## 推荐执行顺序

1. 先修接口主链路混用问题
2. 再修 seed 单一来源问题
3. 再清理未使用依赖和未使用文件
4. 最后再评估数据库迁移

## 本次审计后的直接动作建议

1. 把 `docs/audit/*.md` 作为后续整理基线
2. 先处理首页顶部导航/轮播图残留链路
3. 再做一次真实启动与联调验证
4. 验证通过后，再开始第一批小范围清理
